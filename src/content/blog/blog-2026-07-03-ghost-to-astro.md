一直想搭一个自己的个人博客。

最开始的想法很简单：找一个开源博客系统，部署到自己的服务器上，能够长期写文章、记录项目、整理一些技术笔记。起初我选择的是 Ghost，因为它有后台、有编辑器、界面也比较现代，看起来很适合个人博客和内容发布。

但真正开始安装之后，我发现事情没有想象中那么顺利。

## 最初为什么选择 Ghost

Ghost 给我的第一印象很好。

它不像 WordPress 那样庞大，也不像一些静态博客框架那样完全依赖 Markdown 和命令行。它有完整的后台，可以直接在网页里写文章、管理标签、配置主题。对于一个想认真维护博客的人来说，这种体验是很有吸引力的。

我当时主要看中它几点：

- 有后台管理界面；
- 写作体验比较现代；
- 支持会员、订阅、Newsletter；
- 开源，可以部署在自己的服务器上；
- 相比 WordPress 更轻一些。

所以一开始，我的计划是：

```txt
本地安装 Ghost -> 熟悉后台和主题 -> 之后部署到服务器
```

这个路线看起来很完整，也很适合一个长期维护的个人站点。

## 安装过程中的问题

真正开始安装 Ghost 后，我遇到的第一个问题是环境依赖。

Ghost 对 Node.js、数据库、运行环境都有一定要求。它本身不是一个简单的静态站点，而是一个完整的服务端应用。这意味着我不仅要关心博客内容，还要处理运行环境、数据库、进程管理和后续维护。

在排查过程中，我逐渐意识到几个问题：

- 本地环境和服务器环境需要保持一致；
- 部署后还要考虑服务运行、端口、反向代理等问题；
- 数据库和后台系统会增加维护成本；
- 如果只是个人博客，Ghost 的一些能力暂时用不上。

Ghost 本身没有问题，它依然是一个很成熟的博客系统。但对我当前的需求来说，它有点“重”。

## 重新思考个人博客的需求

卡在安装和部署问题之后，我开始重新想：我到底需要一个什么样的博客？

我真正想要的并不是复杂的后台系统，而是一个稳定、简单、方便维护的内容空间。它需要满足这些目标：

- 可以写技术笔记和项目复盘；
- 页面加载速度快；
- 部署简单；
- 方便自定义样式；
- 代码和内容都能放进 Git 仓库管理；
- 后续可以逐步扩展，而不是一开始就引入太多复杂度。

这样一看，静态站点反而更适合我。

## 为什么最后选择 Astro

后来我把方向转向了 Astro。

Astro 给我的感觉是：它足够轻，也足够现代。对于个人博客来说，它可以把 Markdown 内容直接组织成文章，同时又允许我用组件的方式搭建页面。也就是说，它既保留了静态博客的简单，又给了我不错的前端开发自由度。

选择 Astro 的原因主要有几个：

- 适合内容型网站和个人博客；
- Markdown 写作体验自然；
- 静态构建后部署简单；
- 页面性能好；
- 组件化开发方便后续自定义；
- 项目结构清晰，适合长期维护。

相比 Ghost，Astro 没有后台，也没有数据库。但这反而让我可以把注意力放回内容和页面本身。

## 搭建过程中的调整

在搭建这个博客的过程中，我做了几件个性化调整。

首先是统一个人标识。站点名称、页脚、GitHub 链接都统一成我的代号 LUSH，并把 GitHub 指向自己的账号。

然后是首页和 About 页面。默认模板比较简单，所以我把 Home 页改成了更有个人风格的入口页，让访客一打开网站就能看到我是谁、我在关注什么，以及可以从哪里进入文章和 GitHub。

About 页面则更像个人介绍和阶段经历。我把本科阶段的 Web 软件开发、研究生阶段的 AI 应用开发，以及当前关注的 NLP 与大模型研究整理出来。这样它不只是一个“关于我”的页面，也能说明不同阶段我具体在做什么。

## 当前博客的定位

这个博客现在对我来说有几个作用。

第一，它是一个学习记录空间。之后我会把课程学习、技术笔记、工具实践整理到这里。

第二，它是一个项目复盘空间。做项目时遇到的问题、技术选择、实现过程和总结，都可以沉淀下来。

第三，它也是一个个人展示入口。相比只放一个 GitHub 链接，博客能更完整地呈现我的学习路线、项目思考和长期积累。

## 这次选择带来的体会

这次从 Ghost 转向 Astro，让我重新理解了“适合”比“功能完整”更重要。

Ghost 很强，但它更像一个完整的内容发布系统；Astro 更轻，更适合我当前想要的个人技术博客。对个人站点来说，最重要的不是一开始就拥有所有功能，而是能够持续写下去、维护下去、迭代下去。

所以最终，我选择了 Astro。

这个博客也算正式开始了。
