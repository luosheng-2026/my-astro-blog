在搭建个人博客的同时，我也在整理自己做过的一些项目。

这篇文章记录的是一个本地记账 App 项目：`simplibook`。它也可以叫做 `LUSH Account App`，本质上是一个用 Flutter 开发的 Android 记账应用。

这个项目不复杂，但对我来说比较有代表性。它不是一个单纯的 Demo，而是一个从功能设计、数据存储、页面组织，到最终 APK 编译发布都比较完整的小项目。

## 为什么做这个项目

记账软件其实有很多。

但我想做的是一个足够轻量、足够简单、足够本地化的记账工具。

很多现成记账 App 功能很全，但也往往伴随着账号体系、云同步、广告、会员功能，甚至一些我并不需要的复杂入口。对我来说，最基础的需求其实很明确：

```txt
快速记录一笔收入或支出
能够按时间和分类查看历史账单
能看简单统计
数据保存在本地
可以导出和备份
```

所以 `simplibook` 的定位并不是“大而全”，而是一个小而清晰的本地记账工具。它更像是我对移动端开发、数据持久化和应用发布流程的一次完整练习。

## 项目目标

在开始写代码之前，我给这个项目定了几个目标：

- 使用 Flutter 完成 Android App 开发；
- 支持收入和支出的新增、查看、编辑和删除；
- 使用 SQLite 做本地数据存储；
- 提供基础统计能力；
- 支持数据导出，方便备份；
- 最终能够打包成 APK，在 Android 手机上安装使用。

这些目标看起来都不算复杂，但组合在一起，就已经覆盖了一个本地工具 App 的核心流程。

## 功能设计

这个 App 的功能主要围绕“账单记录”展开。

最核心的功能是新增账单。每一条账单需要包含金额、类型、分类、日期和备注等信息。收入和支出是最基础的区分，分类则用于后续统计，例如餐饮、交通、学习、购物等。

然后是账单列表。用户需要能够按时间顺序查看自己的记录，并快速识别每一笔钱是收入还是支出。列表页看似简单，但它决定了 App 的日常使用体验。

再往后是统计功能。对一个记账 App 来说，统计不一定一开始就要非常复杂，但至少要能看到收入、支出、结余，以及不同分类的大致占比。

最后是导出和备份。因为数据保存在本地，所以导出能力很重要。它能避免用户换设备或重装 App 后完全丢失数据。

## 技术选型

这个项目使用 Flutter 作为主要开发框架。

选择 Flutter 的原因很直接：它适合快速构建跨平台移动端界面，也比较适合我练习完整 App 开发流程。虽然这次主要面向 Android，但 Flutter 的组件体系、状态管理方式和构建工具链都很适合做这种小型应用。

本地数据存储选择 SQLite。

对记账 App 来说，数据结构非常适合关系型数据库：账单、分类、金额、时间这些字段天然可以表结构化。SQLite 又足够轻量，不需要网络服务，也不依赖后端，非常符合“本地化”的定位。

整体技术结构大致可以理解为：

```txt
Flutter UI
  -> 页面与组件
  -> 状态与交互
  -> 本地数据库访问
  -> SQLite 数据持久化
  -> APK 构建与安装
```

## 数据结构思考

记账应用的核心其实是数据模型。

最基础的账单记录可以抽象成一个 `Record`，它至少需要包含这些字段：

- `id`：唯一标识；
- `amount`：金额；
- `type`：收入或支出；
- `category`：分类；
- `date`：日期；
- `note`：备注；
- `createdAt`：创建时间。

这个结构并不复杂，但它决定了后续功能是否容易扩展。例如以后如果想加入预算、周期统计、账单搜索、分类管理，都需要依赖一个稳定的数据模型。

## 页面组织

在页面组织上，我希望它尽量直观。

一个记账 App 不应该让用户在很多层级里来回跳转。常用操作应该尽量短路径完成，所以我倾向于把主要页面拆成几个清晰模块：

- 首页：展示当前收支概况；
- 记账页：新增收入或支出；
- 明细页：查看历史账单；
- 统计页：查看分类和时间维度统计；
- 设置页：导出、备份和一些基础配置。

这样组织之后，用户的路径比较明确：打开 App，看概况；需要记录时进入记账；需要回顾时进入明细或统计。

## 构建过程中的问题

这个项目让我真正接触到了移动端构建流程。

写 Flutter 页面本身是一部分，能不能成功打包 APK 又是另一部分。开发过程中需要处理 Android SDK、Gradle、依赖版本、签名配置等问题。很多时候，代码逻辑没有错，但构建环境会给出各种报错。

比较典型的问题包括：

- 依赖版本不兼容；
- Android Gradle Plugin 版本和 Gradle 版本不匹配；
- SDK 路径或构建工具配置不完整；
- 真机安装时权限或签名问题；
- 打包后资源体积和启动体验需要调整。

这些问题不一定难，但很考验耐心。它们让我意识到：一个项目从“能跑”到“能打包安装”，中间还有一段真实工程距离。

## 项目收获

`simplibook` 对我来说最大的价值，不是它实现了多么复杂的功能，而是让我完整走了一遍本地 App 项目的开发流程。

通过这个项目，我对几个方面有了更具体的理解：

- Flutter 页面和组件如何组织；
- 本地数据库如何设计和访问；
- 移动端应用如何处理状态和数据更新；
- Android APK 如何构建和安装；
- 一个小工具产品如何从需求拆解到功能落地。

它也让我更清楚地意识到，项目经验不是只来自“大项目”。一个小项目只要完整，也能暴露很多真实问题。

## 后续可以改进的方向

如果继续迭代 `simplibook`，我会优先考虑这些方向：

- 增加更清晰的分类管理；
- 优化统计图表；
- 增加按月份、年份筛选账单；
- 支持 CSV 或 JSON 导出；
- 增加本地备份与恢复；
- 改善 UI 细节和交互反馈。

这些功能并不一定要一次性全部做完。对个人项目来说，比较好的节奏是先保证核心流程稳定，再逐步补充体验。

## 总结

`simplibook` 是我的第一个本地记账 App 项目。

它让我从一个具体需求出发，完成了功能设计、数据建模、Flutter 页面开发、本地 SQLite 存储，以及 APK 构建发布等流程。它不是一个复杂产品，但它很完整。

对我来说，这样的项目很适合放进个人博客记录下来。因为它不仅是一个应用，也是一次从想法到落地的工程练习。
