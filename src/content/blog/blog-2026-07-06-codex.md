最近在搭建个人博客和整理项目的过程中，我也开始系统梳理自己使用 Codex 的方式。

一开始使用 Codex 时，我更多是把它当成一个“能读代码、能改代码的 AI 助手”：哪里有问题，就直接把需求丢给它；哪里要改页面，就让它进入项目里帮我调整。

但用得越多，我越发现 Codex 不只是一个对话框。它更像一个可以接入项目上下文、遵守仓库规则、调用工具、复用工作流的开发协作环境。

这篇文章就整理一份我自己的 Codex 指令大全，重点记录：

- `/` 菜单里常用命令分别适合什么场景；
- `AGENTS.md` 和 Skill 应该如何分工；
- 什么时候应该先规划，什么时候可以直接执行；
- 如何把 Codex 变成更稳定的项目开发助手。

## Codex 的两类长期上下文

Codex 的长期行为主要由两类东西控制：`AGENTS.md` 和 Skill。

| 类型 | 作用 | 适合放什么 |
|---|---|---|
| `AGENTS.md` | 项目级或个人级的长期规则 | 构建命令、测试命令、代码规范、目录说明、Review 要求 |
| Skill | 可复用的工作流能力 | 发布流程、PR Review、UI 验收、文档生成、团队 SOP |

简单理解：

- `AGENTS.md` 负责告诉 Codex：在这个项目里一直要遵守什么规则；
- Skill 负责告诉 Codex：遇到某一类任务时，应该按照什么步骤执行。

比如在一个前端项目里，`AGENTS.md` 可以写：

```text
本项目使用 npm。
修改 UI 后需要运行 npm run build。
不要随意引入新的 UI 框架。
```

而 Skill 更适合描述一个固定流程：

```text
当用户要求进行前端验收时：
1. 启动开发服务器；
2. 检查桌面端和移动端布局；
3. 检查可访问性；
4. 检查控制台错误；
5. 输出验收结果。
```

## `/` 菜单常用命令

在 Codex 输入框里输入 `/`，会出现一组可用命令。不同环境、权限和版本下命令可能略有差异，但常见命令大致可以分为几类。

### `/ide`：读取 IDE 上下文

`/ide` 适合让 Codex 读取当前编辑器里的上下文，比如当前打开的文件、选中的代码、光标位置等。

适合场景：

- 解释当前选中的代码；
- 根据正在编辑的文件修 bug；
- 基于当前文件补测试；
- 当前问题和“眼前这段代码”强相关。

示例：

```text
/ide
解释我当前选中的这段代码在做什么
```

```text
/ide
根据当前打开的文件，帮我找出这个报错可能出现在哪里
```

如果问题和具体文件强相关，先让 Codex 看见 IDE 上下文，通常比单纯复制一段代码更稳。

## `/review`：代码审查

`/review` 是我认为最实用的命令之一。

它适合在提交代码前，让 Codex 按代码审查的方式检查当前改动，重点关注 bug、边界条件、破坏性变更和测试缺口。

适合场景：

- 提交代码前；
- 发 PR 前；
- 合并分支前；
- 完成一轮重构后；
- 想检查当前改动有没有潜在风险时。

示例：

```text
/review
审查当前未提交改动，重点关注 bug、边界条件和类型问题
```

也可以要求它按固定格式输出：

```text
请按以下分类输出：
1. 必须修复
2. 建议修复
3. 风格建议
4. 测试建议
```

这类用法很适合放在提交前的最后一步，就像让另一个开发者帮你扫一遍 diff。

## `/plan`：先规划，再动手

如果任务比较复杂，我一般不会一开始就让 Codex 改代码，而是先用 `/plan`。

适合场景：

- 涉及多个文件；
- 需求还不够清晰；
- 可能影响架构；
- 需要先评估改动范围；
- 不希望 Codex 直接修改项目。

示例：

```text
/plan
我想把登录逻辑从组件里抽到 hooks 和 service 层，先给我修改计划，不要直接改代码
```

这样做的好处是，可以先确认方向，再让 Codex 执行。复杂任务里，这一步能减少很多返工。

## `/goal`：持续完成一个目标

`/goal` 更适合完整任务。

它不是只回答一个问题，而是让 Codex 围绕一个目标持续推进，直到完成、被阻塞，或者需要用户补充信息。

适合场景：

- 实现一个完整功能；
- 修复一组相关 bug；
- 重构一个模块；
- 补齐测试；
- 按 PRD 完成一批改动。

示例：

```text
/goal
完成用户登录页重构：保留现有功能，修复移动端样式问题，补充测试，并确保 npm run build 通过
```

我的习惯是：

```text
复杂任务：先 /plan
确认方案后：再 /goal
完成后：最后 /review
```

这套流程比直接让 Codex “帮我实现一下”要稳定得多。

## `/init`：生成项目说明

`/init` 用来初始化项目里的 `AGENTS.md`。

如果一个项目第一次接入 Codex，可以先用它生成一个基础说明文件，然后再补充项目规则。

推荐写入的信息包括：

- 项目技术栈；
- 安装命令；
- 开发命令；
- 构建命令；
- 测试命令；
- 目录结构；
- 代码风格；
- 提交前检查要求。

一个简单示例：

```md
# AGENTS.md

## Project
这是一个 Astro 个人博客项目。

## Commands
- Dev: npm run dev
- Build: npm run build

## Rules
- 不要随意新增依赖。
- 修改页面后运行 npm run build。
- 保持现有路由和内容集合结构。
```

对于长期维护的项目，`AGENTS.md` 非常重要。它可以让 Codex 每次进入项目时都更像“知道项目规矩的协作者”。

## `/mcp`：检查外部工具连接

MCP 可以让 Codex 连接外部工具或共享系统，比如 GitHub、Figma、Linear、内部文档系统等。

`/mcp` 主要用来查看这些连接是否正常。

适合场景：

- Codex 需要读取外部 issue；
- 需要访问设计稿；
- 需要查看 GitHub PR；
- 怀疑某个 MCP 服务没有连接成功。

示例：

```text
/mcp
```

```text
检查 GitHub MCP 是否在线，然后读取这个 PR 的上下文
```

## `/status`、`/model` 和 `/effort`

这三个命令更偏运行状态和能力控制。

| 命令 | 作用 | 适合场景 |
|---|---|---|
| `/status` | 查看当前上下文、会话和额度状态 | 会话很长、担心上下文过多时 |
| `/model` | 切换当前使用的模型 | 简单任务用快模型，复杂任务用强模型 |
| `/effort` | 调整推理强度 | 架构设计、复杂调试、大型 Review |

我的理解是：

- 简单任务不需要一直开最高推理强度；
- 复杂调试、架构设计、代码审查可以提高推理强度；
- 当上下文很长时，可以用 `/status` 判断是否需要开启新会话。

## `/feedback`：反馈产品问题

`/feedback` 不是用来反馈普通代码 bug 的。

它更适合反馈 Codex 自身的问题，例如：

- IDE 上下文读取异常；
- MCP 明明连接了但无法使用；
- Skill 没有出现或无法触发；
- UI、权限、连接状态异常。

如果只是项目代码报错、测试失败、业务逻辑问题，应该直接让 Codex 分析和修复，而不是提交反馈。

## Skill 是什么

Skill 可以理解为 Codex 的可复用工作流。

它通常是一个文件夹，里面至少有一个 `SKILL.md`：

```text
my-skill/
  SKILL.md
  scripts/
  references/
  assets/
```

`SKILL.md` 里至少需要包含 `name` 和 `description`：

```md
---
name: frontend-review
description: Use when reviewing UI changes, especially layout, responsive behavior, accessibility, and console errors.
---

# Frontend Review

1. Start the dev server.
2. Open the affected page.
3. Check desktop and mobile layouts.
4. Check keyboard navigation.
5. Check console errors.
6. Output findings by severity.
```

Skill 可以显式调用，也可以根据描述自动触发。

显式调用通常是：

```text
$frontend-review
检查当前页面的响应式和可访问性
```

## 什么时候应该创建 Skill

适合创建 Skill 的场景：

- 某个流程会反复执行；
- 每次都需要相同步骤；
- 团队有固定 SOP；
- 需要引用模板、脚本或参考资料；
- 希望不同项目里复用同一套流程。

典型 Skill：

| Skill | 用途 |
|---|---|
| `frontend-review` | 前端页面验收 |
| `release-checklist` | 发版前检查 |
| `pr-description` | 生成 PR 描述 |
| `api-docs` | 根据接口代码生成文档 |
| `bug-triage` | 按流程定位 bug |
| `test-case-writer` | 生成测试用例 |

不太适合创建 Skill 的场景：

- 只执行一次的临时任务；
- 只是一条简单规则；
- 只属于当前项目、且每次都应该生效的规则。

这些更适合写进 `AGENTS.md`。

## 全局 Skill 和项目 Skill

Skill 可以分为全局和项目两类。

全局 Skill 放在：

```text
~/.agents/skills
```

适合放个人通用工作流，比如：

- PR 描述生成；
- 日常代码审查；
- 文档润色；
- 发布检查。

项目 Skill 放在：

```text
repo-root/.agents/skills
```

适合放项目专属流程，比如：

- 当前项目的 UI 验收流程；
- 当前项目的发布流程；
- 当前项目的 API 契约检查；
- 当前项目的测试数据生成流程。

我的判断标准是：

| 问题 | 更适合放哪里 |
|---|---|
| 所有项目都可能用 | 全局 Skill |
| 只对当前仓库有效 | 项目 Skill |
| 是稳定项目规则 | `AGENTS.md` |
| 是重复执行的流程 | Skill |
| 是一次性需求 | 普通对话即可 |

## 推荐日常工作流

如果是第一次接入项目：

```text
/init
```

然后补充：

```text
帮我完善 AGENTS.md，加入项目技术栈、目录结构、构建命令、测试命令和代码规范
```

如果是实现复杂功能：

```text
/plan
先分析需求和改动范围，给出实现计划，不要直接改代码
```

确认后：

```text
/goal
按刚才确认的计划实现功能，补充测试，并确保项目检查命令通过
```

完成后：

```text
/review
审查当前改动，重点关注 bug、边界条件、类型问题和破坏性变更
```

如果是发 PR 前：

```text
/review
和 main 分支比较，按严重程度输出问题
```

然后：

```text
$pr-description
根据当前 diff 生成 PR 描述
```

## 总结

Codex 真正好用的地方，不只是“让它帮我改代码”。

更重要的是，把它纳入一个稳定的开发流程里：

- 用 `AGENTS.md` 固化项目规则；
- 用 Skill 复用重复流程；
- 用 `/plan` 降低复杂任务风险；
- 用 `/goal` 推进完整任务；
- 用 `/review` 做提交前检查；
- 用 `/status`、`/model`、`/effort` 管理当前工作状态。

当这些能力组合起来，Codex 就不只是一个问答工具，而更像一个可以长期协作的开发伙伴。
